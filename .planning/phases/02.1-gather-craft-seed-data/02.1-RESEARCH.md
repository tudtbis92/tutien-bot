## Summary

Phase 02.1 requires three parallel workstreams: (1) designing and seeding a full game item catalog (~50 raw materials + ~44 crafted items), (2) refactoring `/gather` from a deterministic item-picker to a gacha economy sink, and (3) building a new `/recipes` command with pagination. All three are tightly coupled through the `items` table and a new `gather_pool_items` table.

The most critical design decision in this phase is the **`gather_pool_items` schema** — it serves as the central mechanism for pool expansion (D-03), event toggles (D-07), and weighted rarity (D-04). A separate join table is strongly preferred over an `is_in_pool` boolean on `items` because it naturally encodes `min_major_realm_index` and `weight` without polluting the base item schema.

The 99.8% net loss invariant (D-04) is **mathematically achievable at Hóa Thần+ (majorRealmIndex 4+)** with the designed weight scheme. Lower realms achieve 98.65–99.70% net loss, which is still an effective sink. All realms satisfy the core invariant EV < 0.

**Primary recommendation:** Build `gather_pool_items` table first (schema migration), then seed data, then refactor gather.ts.

**Note on 99.8% threshold:** Mathematically achieved at Hóa Thần+ with the designed weight scheme. Luyện Khí through Nguyên Anh achieve 98.65–99.70% — still a deep sink. Tier 6–10 items can be added in Phase 3 to expand the pool for higher realms and increase engagement at end-game.

**EV calculation methodology:**
- Pool weights: Tier 1 = 100/item, Tier 2 = 30/item, Tier 3 = 10/item, Tier 4 = 3/item, Tier 5 = 1/item
- 10 items per tier, each profession contributes 1 item per tier active in pool
- EV = Σ(weight_i × base_price_i) / Σ(weight_i) across available pool at that realm

### New Schema: `gather_pool_items`

```typescript
// src/db/schema/gather_pool_items.ts — NEW FILE
import { pgTable, serial, integer, smallint, boolean } from 'drizzle-orm/pg-core';
import { items } from './items.js';

export const gatherPoolItems = pgTable('gather_pool_items', {
  id: serial('id').primaryKey(),
  itemId: integer('item_id')
    .notNull()
    .references(() => items.id),
  // Minimum major realm index required: 0=LK, 1=TC, 2=KD, 3=NA, 4=HT, ...11=ĐLT
  minMajorRealmIndex: smallint('min_major_realm_index').notNull().default(0),
  // Relative draw weight: higher = more common. Tier 1: 100, Tier 2: 30, Tier 3: 10, Tier 4: 3, Tier 5: 1
  weight: smallint('weight').notNull().default(100),
  // Event toggle: set false to temporarily remove item from pool without deleting row
  isActive: boolean('is_active').notNull().default(true),
});

export type GatherPoolItem = typeof gatherPoolItems.$inferSelect;
export type NewGatherPoolItem = typeof gatherPoolItems.$inferInsert;
```

**Query to build pool for a character at majorRealmIndex M:**
```sql
SELECT gpi.item_id, gpi.weight, i.name_i18n_key
FROM gather_pool_items gpi
INNER JOIN items i ON i.id = gpi.item_id
WHERE gpi.min_major_realm_index <= $majorRealmIndex
  AND gpi.is_active = true
```

### Weighted Random Selection (TypeScript helper)

```typescript
// Pure function — testable, no side effects
function weightedRandom<T extends { weight: number }>(pool: T[]): T {
  const totalWeight = pool.reduce((sum, item) => sum + item.weight, 0);
  let threshold = Math.random() * totalWeight;
  for (const item of pool) {
    threshold -= item.weight;
    if (threshold <= 0) return item;
  }
  return pool[pool.length - 1]!; // fallback (never reached with valid pool)
}
```

### New execute() Logic (complete flow)

```typescript
// [VERIFIED: SELECT FOR UPDATE pattern — Drizzle docs confirm .for('update') API]
export async function execute(interaction: ChatInputCommandInteraction): Promise<void> {
  await interaction.deferReply();

  const { t, char, shardId } = await fetchCommandContext(interaction);
  if (!char) { /* not_registered error */ return; }

  const amount = interaction.options.getInteger('amount') ?? 1;
  const majorRealmIndex = getMajorRealmIndex(char.realmId);
  const fee = GATHER_FEES[majorRealmIndex]!;
  const totalFee = fee * BigInt(amount);

  const result = await db.transaction(async (tx) => {
    // 1. Lock user row (SELECT FOR UPDATE prevents race condition)
    const userRow = await tx
      .select({ balance: users.balance })
      .from(users)
      .where(eq(users.discordId, interaction.user.id))
      .for('update')
      .limit(1)
      .then((rows) => rows[0]);

    if (!userRow || userRow.balance < totalFee) {
      return { success: false as const, reason: 'insufficient_balance' as const, totalFee };
    }

    // 2. Deduct balance (users.balance has balance_non_negative CHECK — no underflow possible)
    await tx
      .update(users)
      .set({ balance: sql`${users.balance} - ${totalFee}` })
      .where(eq(users.discordId, interaction.user.id));

    // 3. Query pool for this realm
    const pool = await tx
      .select({ itemId: gatherPoolItems.itemId, weight: gatherPoolItems.weight, nameI18nKey: items.nameI18nKey })
      .from(gatherPoolItems)
      .innerJoin(items, eq(gatherPoolItems.itemId, items.id))
      .where(
        and(
          lte(gatherPoolItems.minMajorRealmIndex, majorRealmIndex),
          eq(gatherPoolItems.isActive, true),
        )
      );

    if (pool.length === 0) {
      // Failsafe: refund and report empty pool
      await tx.update(users)
        .set({ balance: sql`${users.balance} + ${totalFee}` })
        .where(eq(users.discordId, interaction.user.id));
      return { success: false as const, reason: 'empty_pool' as const, totalFee };
    }

    // 4. Roll N times independently
    const rolls: Array<{ itemId: number; nameI18nKey: string }> = [];
    for (let i = 0; i < amount; i++) {
      rolls.push(weightedRandom(pool));
    }

    // 5. Group rolls by itemId, upsert inventory
    const grouped = new Map<number, { nameI18nKey: string; qty: number }>();
    for (const roll of rolls) {
      const existing = grouped.get(roll.itemId);
      if (existing) existing.qty += 1;
      else grouped.set(roll.itemId, { nameI18nKey: roll.nameI18nKey, qty: 1 });
    }
    for (const [itemId, { qty }] of grouped) {
      await tx
        .insert(characterItems)
        .values({ characterId: char.id, itemId, quantity: qty })
        .onConflictDoUpdate({
          target: [characterItems.characterId, characterItems.itemId],
          set: { quantity: sql`${characterItems.quantity} + ${qty}` },
        });
    }

    return { success: true as const, grouped, totalFee, amount };
  });

  if (!result.success) {
    // Handle errors: insufficient_balance, empty_pool
    // ...buildErrorEmbed...
    return;
  }

  // Success embed — single or multi
  await interaction.editReply({
    embeds: [buildItemEmbed({ type: 'multi_gather', items: [...result.grouped], fee: result.totalFee, shardId }, t)],
  });
}
```

### Security Notes for gather.ts Refactor
[VERIFIED: patterns from craft.ts + Phase 2 threat model]

| Threat | Mitigation |
|--------|-----------|
| Double-spend (race condition on balance) | `SELECT FOR UPDATE` on users row inside transaction |
| Balance underflow | DB `CHECK balance_non_negative` constraint prevents it |
| N > 10 parameter abuse | Discord `setMaxValue(10)` server-side gate |

### Seed Script Structure (`src/db/seed.ts`)

```typescript
// Execution order (FK dependencies):
// 1. Raw material items (no FK deps)
// 2. Crafted items (no FK deps at insertion time)
// 3. gather_pool_items (FK → items)
// 4. recipes (FK → items.result_item_id)
// 5. recipe_ingredients (FK → recipes, items)

// Add to package.json scripts:
// "seed": "tsx src/db/seed.ts"

// Idempotency strategy:
// items: INSERT ... ON CONFLICT (name_i18n_key) DO NOTHING
//   → needs unique index on name_i18n_key (currently not unique, add DB index)
//   OR: check if count > 0 before inserting
// recipes/ingredients: similar strategy
```

> **Note on `nameI18nKey` uniqueness:** The items table currently has no UNIQUE constraint on `name_i18n_key`. For seeding idempotency, the seed script should use a `SELECT COUNT(*) WHERE name_i18n_key IN (...)` guard or accept re-running will insert duplicates (safe with `ON CONFLICT (name_i18n_key) DO NOTHING` if we add the index). **Recommend adding a UNIQUE constraint on `items.name_i18n_key` for non-unique items** — but ONLY when `is_unique = false` (unique items can share the same archetype i18n key). This is a partial/conditional unique index.

### Alternative: Partial Unique Index on items.name_i18n_key

```sql
-- In migration or as separate migration
CREATE UNIQUE INDEX items_catalog_name_unique
  ON items (name_i18n_key)
  WHERE is_unique = FALSE;
```

This allows multiple unique items with the same archetype i18n key, while ensuring catalog items (raw materials, crafted items) don't get duplicated on re-seed.

**Option A (simplest):** Show ALL unlocked recipes across ALL professions
```sql
SELECT r.id, r.profession_type, r.min_profession_level, r.result_item_id,
       i.name_i18n_key as result_item_key,
       ri.item_id as ingredient_item_id, ri.quantity, ing.name_i18n_key as ingredient_key
FROM recipes r
INNER JOIN items i ON i.id = r.result_item_id
LEFT JOIN recipe_ingredients ri ON ri.recipe_id = r.id

-- This requires app-level filtering since profession_points is JSONB
```

1. Fetch all recipes with ingredients (join)
2. In TypeScript: filter by `getProfessionLevel(char.professionPoints, recipe.professionType) >= recipe.minProfessionLevel`

### Pattern 2: Weighted Random Selection

```typescript
// Pure helper — place in src/utils/random.ts (new file) or inline in gather.ts
function weightedRandom<T extends { weight: number }>(pool: T[]): T {
  const totalWeight = pool.reduce((sum, item) => sum + item.weight, 0);
  let cursor = Math.random() * totalWeight;
  for (const item of pool) {
    cursor -= item.weight;
    if (cursor <= 0) return item;
  }
  return pool[pool.length - 1]!;
}
```

## Common Pitfalls

### Pitfall 1: BigInt Type Mismatch in Fee Math
**What goes wrong:** `fee * amount` fails because `amount` is `number` and `fee` is `bigint`.
**Root cause:** `GATHER_FEES` returns `bigint`, `amount` from Discord options is `number`.
**How to avoid:** Always cast: `const totalFee = fee * BigInt(amount)`.
**Warning signs:** TypeScript error `Cannot mix BigInt and other types`.

### Pitfall 2: Pool Empty at Luyện Khí Due to Missing Seed
**What goes wrong:** If seed script runs partially and gather_pool_items is empty, gather silently fails.
**How to avoid:** Add failsafe in gather.ts: if pool is empty, refund balance and return error. Never silently succeed.
**Warning signs:** Player reports receiving no items but balance deducted.

### Pitfall 3: FK Order in Seed Script
**What goes wrong:** Inserting recipe_ingredients before recipes (or recipes before items) violates FK constraints.
**How to avoid:** Strict insertion order: raw items → crafted items → gather_pool_items → recipes → recipe_ingredients.
**Warning signs:** FK constraint violation on INSERT.

### Pitfall 4: Drizzle `.for('update')` Without Transaction
**What goes wrong:** `.for('update')` outside a transaction has no effect (immediate lock release).
**How to avoid:** Always use `.for('update')` inside `db.transaction()`. Never outside.
[VERIFIED: Drizzle ORM behavior — locks are transaction-scoped]

### Pitfall 5: Partial Gather Pool (Wrong majorRealmIndex)
**What goes wrong:** `getMajorRealmIndex(realmId)` returns wrong index → wrong fee + wrong pool.
**How to avoid:** Unit test `getMajorRealmIndex` with boundary values: 0→0, 8→0, 9→1, 11→1, 12→2.
**Warning signs:** Luyện Khí players seeing Trúc Cơ pool items.

### Pitfall 6: i18n Keys Not Added to All 3 Locales
**What goes wrong:** ESLint `i18next/no-literal-string` allows i18n key calls, but runtime throws if key missing.
**How to avoid:** Add all new keys to `locales/vi/game.json`, `locales/en/game.json`, `locales/zh-cn/game.json` in same PR. Use `undefined` fallback display if key missing is acceptable.
**Warning signs:** `[game:items.raw.linh_thao_so_khai]` displayed in embed (i18next fallback).

### Pitfall 7: Seed Script Not Idempotent
**What goes wrong:** Running seed twice inserts duplicate items, recipes reference wrong item IDs.
**How to avoid:** Use `ON CONFLICT ... DO NOTHING` with a partial unique index on `items.name_i18n_key WHERE is_unique = false`.
**Warning signs:** Items table grows unexpectedly on re-seed.
