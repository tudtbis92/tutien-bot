**C. src/commands/game/gather.ts — check rows-affected after deduction UPDATE**

In the `db.transaction()` block (around line 153–175), change the balance deduction
UPDATE to capture the Drizzle result and throw if `rowCount === 0`:

```typescript
// BEFORE:
await tx
  .update(characters)
  .set({ tuVi: sql`${characters.tuVi} - ${totalFee}` })
  .where(
    and(
      eq(characters.id, char.id),
      sql`${characters.tuVi} >= ${totalFee}`,
    ),
  );

// AFTER:
const deductResult = await tx
  .update(characters)
  .set({ tuVi: sql`${characters.tuVi} - ${totalFee}` })
  .where(
    and(
      eq(characters.id, char.id),
      sql`${characters.tuVi} >= ${totalFee}`,
    ),
  );
if ((deductResult.rowCount ?? 0) === 0) {
  throw new Error('INSUFFICIENT_BALANCE');
}
```

The `throw` causes Drizzle's transaction wrapper to automatically rollback — no items
are granted if the balance check fails at DB level.

After the transaction, the `INSUFFICIENT_BALANCE` error will propagate to the
`try/catch` in `interactionCreate.ts` and show a generic error. To surface a proper
message, wrap the `db.transaction(...)` call in a try/catch in `gather.ts` itself:

```typescript
try {
  await db.transaction(async (tx) => { ... });
} catch (err) {
  if (err instanceof Error && err.message === 'INSUFFICIENT_BALANCE') {
    await interaction.editReply({
      embeds: [
        buildErrorEmbed(
          t('game:gather.insufficient_balance', {
            required: totalFee.toString(),
            current: char.tuVi.toString(),
          }),
          shardId,
        ),
      ],
    });
    return;
  }
  throw err; // re-throw unexpected errors
}
```

Also, after the successful transaction, compute `remainingBalance` and pass it to
`buildItemEmbed`. Add this after the transaction block:

```typescript
const remainingBalance = char.tuVi - totalFee;
```

Pass `remainingBalance` into both the single-gather `buildItemEmbed(...)` call and
the multi-gather `EmbedBuilder`. For multi-gather, add a footer field or an
additional line to the description: `${t('game:gather.remaining_balance_label')}: **${remainingBalance.toString()}**`.

Note: `char.tuVi` is the value read BEFORE the transaction. `remainingBalance =
char.tuVi - totalFee` is mathematically correct for the post-deduction balance
(assuming no concurrent writes, which the `SELECT FOR UPDATE` in the transaction
covers). This avoids a second DB read just to display the balance.
