# Phase 02.2 Plan 04: Match Cron Jobs & Lifecycle Service Summary

**pg-boss cron automation for full match lifecycle: fixture fetching, odds refreshing, live score polling, and atomic bet resolution with embed posting**

## Accomplishments

- **Match lifecycle service** (`matchLifecycleService.ts`) — 3 exported functions:
  - `postPredictionEmbed`: queries prediction_channels with global/per-league toggle logic, posts embeds via Discord REST API, saves `announcement_message_id`
  - `updateLiveScoreEmbed`: edits existing message via PATCH on score changes, clears prediction components for live/finished matches
  - `resolveMatchBets`: uses `db.transaction()` with `SELECT FOR UPDATE SKIP LOCKED` for atomic, race-condition-safe bet resolution; handles result bets (home/draw/away) and exact score bets; PST/CANC/ABD statuses trigger full refund
- **Football Fetch Fixtures** (`footballFetchFixtures.ts`) — fetches fixtures with `[NOW()+24h, NOW()+7d]` per league via API-Football, skips existing by `fixture_id`, fetches odds, posts prediction embeds
- **Football Refresh Odds** (`footballRefreshOdds.ts`) — refreshes odds for matches kicking off within 24h, only if not updated in last 2 hours, updates embeds with fresh odds
- **Football Poll Scores** (`footballPollScores.ts`) — polls active matches (1H/HT/2H/ET/BT/P) every 15 minutes, batches max 20 per API call, auto-resolves bets on FT/AET/PEN transitions
- **Football Resolve Matches** (`footballResolveMatches.ts`) — resolves finished matches with pending bets, handles stale matches (>2h past kickoff), fetches final scores via API
- **pgBoss.ts** — registers all 4 queues with cron schedules: fetch every 6h, refresh odds every 2h, poll every 15min, resolve every 2h

## Decisions Made

- **REST API over gateway for embeds:** Using `REST.post()`/`REST.patch()` instead of `client.channels.fetch()` avoids shard dependency and is more reliable for background cron jobs
- **Flexible transaction param in resolveMatchBets:** Optional `txDb: typeof db` parameter allows the function to be called either as a standalone transaction or nested inside a caller-provided transaction
- **Score change guard in caller:** The poll job checks for score changes before calling `updateLiveScoreEmbed`, keeping the embed updater function simple
- **4 separate cron jobs (Option C):** Dedicated jobs per lifecycle phase instead of a monolithic handler — clearer error isolation and independent scheduling
