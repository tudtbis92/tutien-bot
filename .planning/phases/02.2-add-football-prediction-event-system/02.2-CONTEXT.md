# Phase 02.2: Add football prediction event system - Context

**Gathered:** 2026-05-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 02.2 adds a football prediction event system where players can bet linh thạch on real football matches. Players predict match results (Win/Draw/Lose) and exact scores as separate bet types, submit via message components, and receive payouts based on real bookmaker odds from API-Football.

**Depends on:** Phase 2 (characters, users table with balance, embed builders, i18n, pg-boss cron infrastructure)

**Not in scope:**
- Live score updates during matches (covered — 15-min polling)
- Other sports betting (football only for v1)
- Social features (leaderboards, prediction sharing)
- Prediction history UI (basic only)

</domain>

<decisions>
## Implementation Decisions

### Prediction Mechanics

- **D-01:** **Two separate prediction types** — match result (Win/Draw/Lose) AND exact score. Each is its own bet with its own payout. Not combined into a single prediction.
- **D-02:** **Message components for submission** — Players interact via buttons/dropdowns on an embed, NOT slash commands. Embed shows match details, odds, and "Predict" buttons.
- **D-03:** **Editable until kickoff** — Players can change their prediction after submitting, as long as the match hasn't started. Once kickoff passes, prediction is locked.

### Wager & Reward System

- **D-04:** **Bet linh thạch** — Players stake real linh thạch from their balance to predict. Not free predictions.
- **D-05:** **Real bookmaker odds from API-Football** — Odds are fetched directly from API-Football and used as-is. No house margin added. Authentic market odds.
- **D-06:** **Balance deducted immediately** — When a player places a bet, linh thạch is deducted from their balance immediately. If they win, payout is credited after resolution.
- **D-07:** **Void bet with full refund** — If the API fails to return a result (match postponed, cancelled, API error), the original bet amount is refunded in full.

### Event Lifecycle & Data Source

- **D-08:** **API-Football as data source** — Use API-Football (api-sports.io) for fixtures, odds, and match results. Free tier: 100 requests/day.
- **D-09:** **pg-boss cron for fetching** — Scheduled jobs fetch upcoming matches, odds, and results from API-Football. Not on-demand.
- **D-10:** **Predictions open 24h before kickoff** — Predictions become available 24 hours before match starts. Lock exactly at kickoff time.
- **D-11:** **Embed announcements in dedicated channel** — Bot posts formatted embeds with match details, odds, and prediction buttons to a specific channel (e.g., `#predictions`).
- **D-12:** **Auto-resolve via cron** — After match finishes, a cron job fetches the result, calculates winners, and distributes payouts automatically.

### Integration with Existing Systems

- **D-13:** **No tu vi accumulation** — Prediction activity does NOT count toward tu vi. Purely an economy feature, separate from cultivation progression.
- **D-14:** **No impact on streaks or character stats** — Winning/losing predictions does not affect streak days or any other character attributes.

### API Usage Management

- **D-15:** **Curated league list** — Only fetch matches from these leagues: Premier League, La Liga, Ligue 1, Bundesliga, Serie A, Champions League, Europa League, Conference League, World Cup, Euro.
- **D-16:** **All matches from selected leagues** — Create prediction events for every match in the curated leagues. No daily cap.
- **D-17:** **5-key pool for fallback/rotation** — Support multiple API keys stored in config/DB. Round-robin rotation with skip on 429. ⚠️ **ToS risk:** API-Football ToS forbids multiple free accounts. Design key pool as fallback/reliability system that can easily reduce to single paid key later.
- **D-18:** **15-min live score polling** — During active matches with predictions, poll API every 15 minutes for score updates. Update embed with current score. Match end triggers auto-resolution.
- **D-19:** **DB caching for API responses** — Store API responses in PostgreSQL to avoid re-fetching the same data.

### the agent's Discretion

- API-Football endpoint selection (which v3 endpoints for fixtures, odds, fixtures/events)
- Database schema design for predictions, bets, matches tables
- Embed layout and button design (follow existing embed builder patterns)
- Key rotation implementation details (round-robin vs quota-aware)
- Error handling and retry logic for API failures
- i18n keys for all match-related strings, team names, league names
- Bet minimum/maximum limits (agent should propose reasonable defaults)
- Payout calculation precision (decimal handling for odds multiplication)

### Post-Planning Decisions

- **D-20:** **Per-channel prediction config** — `/config predictions` uses `interaction.channelId` (channel where command is invoked) as target, not guild-level. Schema: `prediction_channels` (guild_id + channel_id unique, enabled boolean) + `prediction_channel_leagues` (channel_id + league_id unique, enabled boolean). Admin can enable/disable predictions per channel and toggle leagues per channel.
- **D-21:** **Option C fetch lifecycle** — 4 cron jobs: (1) `footballFetchFixtures` every 6h — fetch fixtures with kickoff in [NOW()+24h, NOW()+7d] for prediction opening; (2) `footballRefreshOdds` every 2h — refresh odds for fixtures with kickoff in [NOW(), NOW()+24h]; (3) `footballPollScores` every 15min — poll live scores; (4) `footballResolveMatches` every 2h — resolve finished matches.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/ui/embeds/` — Existing embed builders (buildSuccessEmbed, buildErrorEmbed, buildItemEmbed). New prediction embeds must follow same pattern.
- `src/utils/commandContext.ts` — `fetchCommandContext()` for locale resolution, character lookup. Prediction commands will use this.
- `src/i18n/index.ts` — `resolveLocale()`, `getT()` for i18n. All prediction strings must use i18n keys.
- `src/jobs/` — Existing pg-boss job patterns. Prediction fetch/resolve jobs will follow same structure.
- `src/db/schema/` — Drizzle ORM schema patterns. New tables (predictions, bets, matches) follow same conventions.

### Established Patterns
- **Atomic transactions**: `db.transaction()` for any balance changes — prediction bets and payouts must use this.
- **Embed builders**: Typed params, shard-aware, i18n-integrated. Prediction embeds follow same.
- **Command autodiscovery**: Drop into `src/commands/game/` — auto-loaded.
- **i18n enforcement**: ESLint rule blocks hardcoded strings — all prediction strings via i18n keys.
- **BIGINT currency**: `users.balance` is BIGINT, never float. All bet amounts and payouts use BIGINT arithmetic.

### Integration Points
- `users.balance BIGINT` — Bet deduction and payout crediting. Need `SELECT FOR UPDATE` to prevent race conditions.
- `pg-boss` — Cron jobs for match fetching, live score polling, result resolution.
- `i18n` — All user-facing strings (match announcements, bet confirmations, result notifications) need VI/EN/ZH-CN keys.
- New tables needed: `football_matches`, `football_predictions`, `football_bets` (or similar naming).

</code_context>
