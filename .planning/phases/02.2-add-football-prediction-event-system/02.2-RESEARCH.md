# Phase 02.2: Add football prediction event system - Research

**Researched:** 2026-05-20
**Domain:** API-Football v3 integration, Discord message components, prediction economy
**Confidence:** HIGH

## Summary

This phase adds a football prediction event system where players bet linh thạch on real football matches using API-Football (api-sports.io) data. Players predict match results (Win/Draw/Lose) and exact scores as separate bet types via Discord message component buttons on embeds. Predictions are editable until kickoff, bets deduct balance immediately, and payouts use real bookmaker odds fetched from the API. The system uses pg-boss cron jobs for match fetching, 15-minute live score polling during active matches, and auto-resolution after match completion.

**Primary recommendation:** Use API-Football v3 with a curated league list, DB-cached fixtures/odds, pg-boss scheduled jobs for all API interactions, and Discord message components (StringSelectMenu + Buttons) for prediction submission. Follow existing embed builder, i18n, and BIGINT transaction patterns.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Two separate prediction types — match result (Win/Draw/Lose) AND exact score. Each is its own bet with its own payout. Not combined into a single prediction.
- **D-02:** Message components for submission — Players interact via buttons/dropdowns on an embed, NOT slash commands. Embed shows match details, odds, and "Predict" buttons.
- **D-03:** Editable until kickoff — Players can change their prediction after submitting, as long as the match hasn't started. Once kickoff passes, prediction is locked.
- **D-04:** Bet linh thạch — Players stake real linh thạch from their balance to predict. Not free predictions.
- **D-05:** Real bookmaker odds from API-Football — Odds are fetched directly from API-Football and used as-is. No house margin added. Authentic market odds.
- **D-06:** Balance deducted immediately — When a player places a bet, linh thạch is deducted from their balance immediately. If they win, payout is credited after resolution.
- **D-07:** Void bet with full refund — If the API fails to return a result (match postponed, cancelled, API error), the original bet amount is refunded in full.
- **D-08:** API-Football as data source — Use API-Football (api-sports.io) for fixtures, odds, and match results. Free tier: 100 requests/day.
- **D-09:** pg-boss cron for fetching — Scheduled jobs fetch upcoming matches, odds, and results from API-Football. Not on-demand.
- **D-10:** Predictions open 24h before kickoff — Predictions become available 24 hours before match starts. Lock exactly at kickoff time.
- **D-11:** Embed announcements in dedicated channel — Bot posts formatted embeds with match details, odds, and prediction buttons to a specific channel (e.g., `#predictions`).
- **D-12:** Auto-resolve via cron — After match finishes, a cron job fetches the result, calculates winners, and distributes payouts automatically.
- **D-13:** No tu vi accumulation — Prediction activity does NOT count toward tu vi. Purely an economy feature, separate from cultivation progression.
- **D-14:** No impact on streaks or character stats — Winning/losing predictions does not affect streak days or any other character attributes.
- **D-15:** Curated league list — Only fetch matches from these leagues: Premier League, La Liga, Ligue 1, Bundesliga, Serie A, Champions League, Europa League, Conference League, World Cup, Euro.
- **D-16:** All matches from selected leagues — Create prediction events for every match in the curated leagues. No daily cap.
- **D-17:** 5-key pool for fallback/rotation — Support multiple API keys stored in config/DB. Round-robin rotation with skip on 429. ⚠️ **ToS risk:** API-Football ToS forbids multiple free accounts. Design key pool as fallback/reliability system that can easily reduce to single paid key later.
- **D-18:** 15-min live score polling — During active matches with predictions, poll API every 15 minutes for score updates. Update embed with current score. Match end triggers auto-resolution.
- **D-19:** DB caching for API responses — Store API responses in PostgreSQL to avoid re-fetching the same data.

### Pattern 1: API Client with Key Rotation
**What:** HTTP client for API-Football that manages a pool of API keys, rotates on 429, and tracks per-key quota.

```typescript
// Source: [ASSUMED] — pattern design based on D-17 requirements
import { config } from '../config.js';
import { logger } from '../utils/logger.js';

export interface ApiKeyEntry {
  key: string;
  remaining: number;    // x-ratelimit-requests-remaining from last response
  last429At: number | null; // timestamp of last 429
}

// Key pool loaded from config — designed to work with single key
// ToS: multiple free accounts forbidden — pool is for reliability, not quota evasion
const keyPool: ApiKeyEntry[] = config.API_FOOTBALL_KEYS.split(',').map((k) => ({
  key: k.trim(),
  remaining: 100,
  last429At: null,
}));

let currentKeyIndex = 0;

export async function apiFootballFetch(path: string, params: Record<string, string> = {}): Promise<any> {
  const maxRetries = keyPool.length;
  let attempts = 0;

  while (attempts < maxRetries) {
    const keyEntry = keyPool[currentKeyIndex];
    const url = new URL(`https://v3.football.api-sports.io${path}`);
    Object.entries(params).forEach(([k, v]) => url.searchParams.set(k, v));

    const response = await fetch(url.toString(), {
      headers: {
        'x-apisports-key': keyEntry.key,
      },
    });

    // Update quota tracking from response headers
    const remaining = response.headers.get('x-ratelimit-requests-remaining');
    if (remaining) keyEntry.remaining = parseInt(remaining, 10);

    if (response.status === 429) {
      keyEntry.last429At = Date.now();
      logger.warn('FootballAPI', `Key ${currentKeyIndex} rate limited — rotating`);
      currentKeyIndex = (currentKeyIndex + 1) % keyPool.length;
      attempts++;
      await new Promise((r) => setTimeout(r, 1000 * attempts)); // backoff
      continue;
    }

    if (!response.ok) {
      throw new Error(`API-Football ${response.status}: ${response.statusText}`);
    }

    const data = await response.json();
    if (data.errors && Object.keys(data.errors).length > 0) {
      throw new Error(`API-Football errors: ${JSON.stringify(data.errors)}`);
    }

    return data;
  }

  throw new Error('API-Football: all keys rate limited');
}
```

### Pattern 2: BIGINT-Safe Payout Calculation
**What:** Multiply BIGINT balance by decimal odds without floating-point precision loss.

```typescript
// Source: [ASSUMED] — pattern design for D-05 + D-06
/**
 * Calculate payout: betAmount × odds, rounded down to integer.
 * Uses integer math to avoid floating-point precision issues with BIGINT.
 *
 * @param betAmount - Wagered linh thạch (BIGINT)
 * @param decimalOdds - Bookmaker odds as string (e.g., "2.50")
 * @returns Payout amount (BIGINT) — includes original stake
 */
export function calculatePayout(betAmount: bigint, decimalOdds: string): bigint {
  // Parse odds as integer × 10000 to preserve 4 decimal places
  const oddsInt = Math.round(parseFloat(decimalOdds) * 10000);
  // betAmount × oddsInt / 10000 — integer division truncates (rounds down)
  return (betAmount * BigInt(oddsInt)) / 10000n;
}

// Example: 1000 linh thạch × 2.50 odds = 2500 payout
// calculatePayout(1000n, "2.50") → 2500n
```

### Anti-Patterns to Avoid
- **Floating-point odds math:** Never use `Number(balance) * odds` — loses precision on BIGINT values. Use integer math (Pattern 2).
- **On-demand API calls on interaction:** D-09 requires pg-boss cron, not fetch-on-click. All API data must be pre-fetched and cached.
- **Storing API keys in .env only:** D-17 requires a key pool in config/DB for rotation. Design for easy reduction to single key.
- **Hardcoded team/league names:** i18n enforcement — all display strings must use i18n keys.
- **Race condition on balance:** Always use `SELECT FOR UPDATE` or atomic `UPDATE ... WHERE balance >= amount` for bet deduction.
- **Duplicate embed posting:** Track `announcement_message_id` per match to update existing embed (live scores) instead of posting new ones.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP client for API-Football | Custom retry/rotation logic from scratch | Native `fetch` + structured key pool service | Node 22 fetch is production-ready; only rotation logic is custom |
| Decimal odds math | `parseFloat` + `Number` multiplication | Integer math with 10000× scaling | Floating-point loses precision on large BIGINT balances |
| Message component interaction handling | Raw event listener on `interactionCreate` | Discord.js `ComponentType.StringSelect` / `Button` handlers | Type-safe, auto-parses customId |
| Cron job scheduling | `setInterval` or `setTimeout` | pg-boss `schedule()` | Survives restarts, distributed-safe, already in project |
| Embed building | Inline `new EmbedBuilder()` in every handler | `buildPredictionEmbed()` in `src/ui/embeds/` | Consistent styling, follows existing pattern |
| API response caching | In-memory Map or variable | PostgreSQL `api_cache` table | Survives restarts, shard-safe (D-19) |

**Key insight:** The only truly custom pieces are the API key rotation service and the prediction embed builder. Everything else reuses existing project patterns (pg-boss, Drizzle transactions, embed builders, i18n).

## Common Pitfalls

### Pitfall 1: API-Football 7-Day Odds History Limit
**What goes wrong:** Odds are only available for the last 7 days. If you fetch fixtures 14 days before kickoff, odds won't be available yet.
**Why it happens:** API-Football design — odds data is not stored indefinitely.
**How to avoid:** Fetch fixtures early (store fixture metadata), but fetch odds only within 7 days of kickoff. Separate fixture fetch job from odds fetch job.
**Warning signs:** `/odds?fixture=X` returns empty `bookmakers` array for fixtures > 7 days out.

### Pitfall 2: Free Tier 100 Requests/Day Exhaustion
**What goes wrong:** 10 curated leagues × multiple matchdays × odds + fixtures + polling = easily exceeds 100/day.
**Why it happens:** Each API call counts. Live polling 15-min × N active matches burns quota fast.
**How to avoid:** 
- Fetch fixtures once per day per league (10 calls)
- Fetch odds once per day per league with active fixtures (~10 calls)
- Live polling: only poll matches with active bets, batch via `/fixtures?ids=` (max 20 per call)
- Cache aggressively in DB — never re-fetch same data
- **Recommendation:** Pro plan ($19/mo, 7500 req/day) is the practical minimum for production
**Warning signs:** `x-ratelimit-requests-remaining` header drops below 10.

### Pitfall 3: Odds Pagination (10 per page)
**What goes wrong:** `/odds` endpoint returns 10 results per page. League-wide odds queries need pagination.
**Why it happens:** API design for large response sets.
**How to avoid:** Use `?fixture=` parameter (single fixture, no pagination) instead of `?league=` + `?season=`. Fetch odds per-fixture when announcing each match.
**Warning signs:** `paging.total > 1` in API response — means you missed data.

### Pitfall 4: Match Status Edge Cases (Postponed/Cancelled)
**What goes wrong:** Matches get postponed (PST), cancelled (CANC), or abandoned (ABD). Resolution logic must handle these.
**Why it happens:** Real-world events — weather, security, etc.
**How to avoid:** D-07 requires void bet with full refund. Resolution job must check for PST/CANC/ABD statuses and refund all bets.
**Warning signs:** Fixture status is not FT/AET/PEN after expected end time.

### Pitfall 5: API-Football ToS — Multiple Free Accounts Forbidden
**What goes wrong:** D-17 specifies 5-key pool. API-Football ToS explicitly forbids multiple free accounts.
**Why it happens:** ToS: "It is forbidden to have multiple accounts to increase the limit of the free plan."
**How to avoid:** Design key pool as reliability system (works with single paid key). If using free tier, use only 1 key. Key pool rotation handles paid key rotation (multiple keys from same account via dashboard member invites).
**Warning signs:** Account suspension risk if multiple free accounts detected.

### Pitfall 6: Embed Edit Rate Limits
**What goes wrong:** Updating live score embeds every 15 minutes for many matches hits Discord rate limits.
**Why it happens:** Discord API has edit rate limits per channel.
**How to avoid:** Track `announcement_message_id` per match. Edit existing message (not post new). Batch edits with delays. Skip updates if score hasn't changed.
**Warning signs:** Discord 429 on `PATCH /channels/{id}/messages/{msgId}`.

## Open Questions

1. **Exact Score odds availability**
   - What we know: API-Football has `/odds/bets` endpoint listing all bet types. Correct Score is a standard bet type.
   - What's unclear: Whether Correct Score odds are available for ALL curated leagues or only top-tier ones. Coverage varies by competition.
   - Recommendation: First implementation should fetch `/odds/bets` at startup, cache bet IDs, and gracefully skip exact score predictions when Correct Score odds are unavailable for a fixture.

2. **Bet minimum/maximum limits**
   - What we know: D-04 requires betting linh thạch. No limits specified in decisions.
   - What's unclear: What are reasonable defaults?
   - Recommendation: Minimum 100 linh thạch (meaningful bet, prevents spam). Maximum 1,000,000 linh thạch (prevents catastrophic loss). Make configurable.

3. **Channel configuration for predictions**
   - What we know: D-11 requires dedicated channel. No admin command exists yet.
   - What's unclear: How does admin configure which channel?
   - Recommendation: Store `guild_id → channel_id` mapping in a new `guild_settings` table or add to existing config. Admin sets via `/config predictions-channel #channel`.

4. **Season year calculation for leagues**
   - What we know: API-Football uses starting year (2025/2026 season = 2025).
   - What's unclear: How to determine current season for each league programmatically?
   - Recommendation: Call `/leagues?id=X` once at startup, extract `seasons` array, use last element. Cache result.

### Known Threat Patterns for Football Prediction System

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Bet placement race condition (double-spend) | Tampering | `UPDATE users SET balance = balance - $amount WHERE discord_id = $1 AND balance >= $amount` — atomic check |
| Prediction edit after kickoff (time manipulation) | Tampering | Server-side kickoff time check on every edit attempt |
| API key exposure in logs/code | Information Disclosure | Keys in env vars only, never logged, redacted in error messages |
| Malicious fixture ID in component customId | Spoofing | Validate fixtureId exists in DB before processing |
| Negative bet amount via crafted input | Tampering | Zod validation: `z.coerce.bigint().positive()` |
| API response manipulation (MITM) | Tampering | HTTPS-only, validate response structure before processing |
| Excessive bet amount draining user | Tampering | Configurable max bet limit enforced server-side |
