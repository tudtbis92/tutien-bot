    **IMPORTANT DESIGN: Redesigned interaction flow**
    Each selection (result dropdown OR score button) directly triggers a Modal for wager amount → bet placed.
    This eliminates the need for shared ephemeral state between steps. D-01 says each bet type is separate,
    so 2 independent flows align perfectly.

    **Flow A (Result bet):** User selects from dropdown → Modal asks wager → bet placed
    **Flow B (Score bet):** User clicks score button → Modal asks wager → bet placed

    Create `src/services/football/predictionService.ts`:
    - Extract ALL business logic from component handlers for testability without Discord context.
    - Export async function `placeBet(db, userId: number, matchId: number, betType: 'result' | 'score', prediction: string, wagerAmount: bigint)`: 
      - Validate match exists and status = 'NS', kickoff_at > NOW() (per D-03, D-10).
      - Validate wager via validateBetAmount() (per FOOTBALL_CONFIG.MIN_BET/MAX_BET).
      - Look up appropriate odds from football_matches (homeOdds/drawOdds/awayOdds for result, exactScoreOdds for score).
      - Calculate potential payout via calculatePayout(wager, odds).
      - Check if user already has a pending bet for (userId, matchId, betType) — if yes, EDIT flow:
        - Refund old wager to user balance.
        - Mark old bet as void.
      - Execute atomic transaction (per D-06):
        1. SELECT user balance FOR UPDATE.
        2. Verify balance >= wager. If not, throw InsufficientBalanceError.
        3. UPDATE users SET balance = balance - wager WHERE id = user_id AND balance >= wager.
        4. INSERT (or UPDATE if editing) football_bets with matchId, betType, prediction, wagerAmount, potentialPayout, oddsUsed, status='pending'.
      - Return { bet, payout, isEdit } for handler to format response.
      - Per D-13, D-14: do NOT touch tu_vi, streak_days, or any character stats.
    - Export async function `getUserBets(db, userId: number, status?: string)`: fetch user's bets joined with match data.
    - NOTE: football_bets.fixture_id FK → football_matches.id (internal PK). matchId parameter is the internal PK.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Discord component → handler | User submits prediction — must validate match state, user identity, bet amount |
| Modal input → predictionService | Wager amount from user input — Zod validate before processing |
| Handler → users.balance | Atomic deduction prevents race conditions and double-spend |
| Edit prediction → old bet refund | Refund must be in same transaction as new bet placement |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.2-10 | Tampering | Bet after kickoff | mitigate | Server-side NOW() >= kickoff_at check in predictionService.placeBet() |
| T-02.2-11 | Tampering | Double-spend on bet placement | mitigate | `UPDATE users SET balance = balance - $w WHERE balance >= $w` — atomic, returns 0 rows if insufficient |
| T-02.2-12 | Tampering | Bet amount via crafted modal input | mitigate | Zod validation on modal submit: `z.coerce.bigint().positive()`, check MIN/MAX bounds |
| T-02.2-13 | Spoofing | Fake matchId in customId | mitigate | Lookup matchId in football_matches table — if not found, reject immediately |
| T-02.2-14 | Tampering | Edit someone else's bet | mitigate | WHERE clause includes user_id — can only UPDATE own bet rows |
| T-02.2-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages required |
</threat_model>
