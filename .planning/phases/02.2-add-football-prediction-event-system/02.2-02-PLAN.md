    - Private method `getNextKey()`: round-robin key selection from pool. Parse `config.API_FOOTBALL_KEYS` split by comma, trim whitespace. Track per-key remaining quota and last429 timestamp in memory Map. Skip keys with last429 within 60 seconds.
    - Private method `getCache(key: string)`: query api_cache table where cache_key = key AND expires_at > NOW(). Return cached response if found.
    - Private method `setCache(key: string, endpoint: string, data: any, ttlMinutes: number)`: insert into api_cache with expires_at = NOW() + ttlMinutes. Use ON CONFLICT (cache_key) DO UPDATE.
    - Public async method `fetch(path: string, params: Record<string, string>, ttlMinutes: number = 60)`: 
      1. Build cache key from path + sorted params hash (use simple string concatenation: `${path}?${sortedParams}`).
      2. Check cache first — return cached data if valid.
      3. Select next API key from pool.
      4. Call Node 22 native `fetch` with URL `https://v3.football.api-sports.io${path}` and headers `{ 'x-apisports-key': key }`.
      5. On 429: mark key as rate-limited, rotate to next key, retry with 1s backoff × attempt count. Max retries = key pool length.
      6. On non-OK response: throw Error with status + statusText.
      7. Parse JSON response. If `data.errors` has keys, throw Error with error details.
      8. Cache the response with TTL.
      9. Return `data.response` (the API wraps data in `response` array).
    - Convenience methods: `getFixtures(leagueId, season)`, `getFixtureOdds(fixtureId)`, `getFixtureScores(fixtureIds: number[])`, `getFixtureResult(fixtureId)` — each calls `fetch()` with appropriate path and params.
    - Use `CURATED_LEAGUES` for league filtering. Use `FOOTBALL_CONFIG.MAX_BATCH_FIXTURE_IDS` for score batching (slice to 20 max, join with '-').
    - All API calls use native Node 22 `fetch` — no external HTTP client.
